## 🔴 高优先级问题（必须修复）

### 1. 绑定码安全性严重不足 ⛔

**位置:** `GuardianStarApp.java:106`, `GuardianStarApp.java:106`

```java
// ❌ 问题代码
String bindCode = String.valueOf((int) (Math.random() * 900000) + 100000);
```

**风险说明:**
- `Math.random()` 使用 `java.util.Random`，是伪随机数生成器
- 随机种子基于系统时间，在已知大致时间范围内可预测
- 6 位数字有 900,000 种可能，暴力破解只需数小时
- **攻击方式:** 通过 `ANDROID_ID` 获取设备后，批量尝试绑定码

**修复建议:**
```kotlin
// ✅ 安全实现
val secureRandom = SecureRandom()
val bytes = ByteArray(3)
secureRandom.nextBytes(bytes)
val bindCode = (bytes[0].toInt().and(0xFF) * 1000 + bytes[1].toInt().and(0xFF) * 10 + bytes[2].toInt().and(0xFF))
    .toString().padStart(6, '0')
```

### 2. 数据库无迁移逻辑，数据将丢失 ⛔

**位置:** `AppDatabase.java:42`

```java
// ❌ 问题代码
.fallbackToDestructiveMigration()
```

**风险说明:**
- `fallbackToDestructiveMigration()` 在版本不匹配时直接删除并重建数据库
- 用户所有应用使用记录、定时设置、应用限制数据**全部丢失**
- `version = 2` 意味着任何 schema 变更都会触发清空
- 无 `migration()` 实现，无法保留历史数据

**修复建议:**
```kotlin
// ✅ 实现真实迁移
val MIGRATION_1_2 = object : Migration(1, 2) {
    override fun migrate(database: SupportSQLiteDatabase) {
        // 示例: 新增字段保留数据
        database.execSQL("ALTER TABLE devices ADD COLUMN last_report_time INTEGER")
    }
}
.fallbackToDestructiveMigrationFrom(1, MIGRATION_1_2)
```

### 3. 内存泄漏：单例持有 Context ⛔

**位置:** `UsageMonitorService.java:16-22`, `GuardianStarApp.java:22`

```java
// ❌ 问题代码
private static UsageMonitorService instance;
public static UsageMonitorService getInstance() { return instance; }

private static GuardianStarApp instance;
public static GuardianStarApp getInstance() { return instance; }
```

**风险说明:**
- `UsageMonitorService` 的静态 `instance` 持有了 Service 对象引用
- Service 持有 Application Context，但静态引用阻止 GC 回收
- 服务销毁时（`onDestroy()`）设置 `instance = null`，但仍有短暂窗口期泄漏
- `GuardianStarApp.getInstance()` 返回 Application 本身（无泄漏，但设计不优雅）

**修复建议:**
```kotlin
// ✅ 使用 WeakReference 或改为非静态访问
private var instance: UsageMonitorService? = null
fun getInstance(): UsageMonitorService? = instance
// 或在 Service 内部访问，不对外暴露静态引用
```

### 4. 字符串拼接构建 SharedPreferences key 存在风险 ⛔

**位置:** `GuardianService.java:315`, `UsageMonitorService.java:71`

```java
// ❌ 问题代码
editor.putLong("limit_" + packageName, dailyLimitMinutes);
long limitMinutes = prefs.getLong("limit_" + packageName, 0);
```

**风险说明:**
- `packageName` 来自用户输入/服务端下发，未做校验
- 恶意设备发送包含特殊字符的包名（如 `.`、`;`、`\n`）可能导致 key 冲突
- 超长包名（某些设备可达 255 字符）可能导致存储异常
- 无长度限制，服务端可下发 1000+ 个限制，导致SharedPreferences 膨胀

**修复建议:**
```kotlin
// ✅ 加校验 + 哈希化处理
val packageNameHash = packageName.sha256().take(16)
editor.putLong("limit_$packageNameHash", dailyLimitMinutes)
```

### 5. 儿童端锁定功能在 Android 10+ 无效 ⛔

**位置:** `LockActivity.java:91-96`

```java
// ❌ 问题代码
if (keyCode == KeyEvent.KEYCODE_HOME || keyCode == KeyEvent.KEYCODE_APP_SWITCH) {
    return true;
}
```

**风险说明:**
- Android 10 (API 29) 之后，普通应用**无法拦截** HOME 键和 APP_SWITCH（多任务键）
- 即使拦截返回 `true`，系统仍会正常处理
- 孩子可以通过 Home 键或最近任务离开锁定界面
- 家长可能误以为设备已锁定，实际未生效

**修复建议:**
```kotlin
// ✅ 使用 DeviceAdmin/DeviceOwner 模式
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.LOLLIPOP) {
    val devicePolicyManager = getSystemService(Context.DEVICE_POLICY_SERVICE) as DevicePolicyManager
    if (devicePolicyManager.isDeviceOwnerApp(packageName)) {
        // 可以设置锁定任务模式
        startLockTask()
    }
}

// 或使用辅助功能模拟返回/Home 操作（有限制）
val gesture = AccessibilityService.GestureDescription.Builder()
    .addStroke(GestureDescription.StrokeDescription(Rect(), 0, 100))
    .build()
dispatchGesture(gesture, null, null)
```

### 6. 网络请求重试使用 Thread.sleep 阻塞 ⛔

**位置:** `ApiClient.java:78`

```java
// ❌ 问题代码
try { Thread.sleep(waitMs); } catch (InterruptedException ignored) {}
```

**风险说明:**
- `Thread.sleep()` 阻塞当前线程，在 OkHttp 的异步回调中调用会导致 UI 卡顿
- 如果在主线程调用（例如在某些回调链中），会造成 ANR
- 没有指数退避，固定 1000ms * attempt，网络波动时重试节奏不合理

**修复建议:**
```kotlin
// ✅ 使用 OkHttp 内置重试 + 退避
val client = OkHttpClient.Builder()
    .retryOnConnectionFailure(true)
    .callTimeout(60, TimeUnit.SECONDS)
    .addInterceptor(RetryWithBackoffInterceptor())
    .build()

// RetryWithBackoffInterceptor 使用指数退避：1s, 2s, 4s, 8s...
```

### 7. SOS 触发静默无通知 ⛔

**位置:** `SOSReceiver.java:33-38`

```java
private void triggerSOS(Context context) {
    Vibrator vibrator = (Vibrator) context.getSystemService(Context.VIBRATOR_SERVICE);
    if (vibrator != null) {
        vibrator.vibrate(500);  // ⚠️ 仅振动500ms，无声音无通知
    }
}
```

**风险说明:**
- SOS 是紧急场景，仅振动不足以提醒用户
- 无声音提示，在静音模式下完全无感知
- 无弹窗通知，孩子可能不知道 SOS 已触发
- 无网络上报机制（代码中未见实际网络触发）

**修复建议:**
```kotlin
// ✅ 完整 SOS 触发
val vibrator = context.getSystemService(Context.VIBRATOR_SERVICE) as Vibrator
vibrator.vibrate(VibrationEffect.createOneShot(2000, VibrationEffect.DEFAULT_AMPLITUDE))

// 播放紧急声音
val audioManager = context.getSystemService(Context.AUDIO_SERVICE) as AudioManager
audioManager.playSoundEffect(AudioManager.SOUND_EFFECT_1)

// 发送网络 SOS 警报（实际代码中缺少此部分）
```

### 8. Service 生命周期耦合：GuardianService → UsageMonitorService

```java
java.util.Map<String, Long> allUsage = UsageMonitorService.getInstance() != null ?
```

### 9. MainActivity 每次 onResume 都重新请求 API

**位置:** `MainActivity.java:169-172`

```java
@Override
protected void onResume() {
    super.onResume();
    loadDashboardFromServer();  // ⚠️ 每次返回都重新请求
}
```

**问题:**
- 用户从子页面返回主界面时，每次都发起网络请求
- 频繁请求可能导致：
  - 流量浪费
  - 服务器压力增大
  - 多个请求并发（快速返回时）
- 无请求缓存或防抖机制

**修复建议:**
```kotlin
// ✅ 添加防抖 + 缓存
private var lastRefreshTime = 0L
private val REFRESH_INTERVAL = 5 * 60 * 1000L // 5分钟

override fun onResume() {
    super.onResume()
    val now = System.currentTimeMillis()
    if (now - lastRefreshTime > REFRESH_INTERVAL) {
        loadDashboardFromServer()
        lastRefreshTime = now
    }
}
```

### 11. WebSocket 心跳 Timer 泄漏风险

**位置:** `WebSocketService.java:366-384`, `GuardianService.java:368-383`

```java
// ❌ 问题代码
heartbeatTimer = new Timer();
heartbeatTimer.scheduleAtFixedRate(new TimerTask() { ... }, 30000, 30000);
```

**问题:**
- `Timer` 是单线程的，任何任务异常都会导致后续任务停止
- `TimerTask` 异常未捕获时，整个 Timer 终止，心跳停止
- 未检查 `TimerTask` 是否已取消就调度新任务
- `Timer` 内部线程持有对 Task 的引用，可能导致内存泄漏

**修复建议:**
```kotlin
// ✅ 使用 ScheduledExecutorService 或 Handler
private val heartbeatHandler = Handler(Looper.getMainLooper())
private val heartbeatRunnable = Runnable {
    if (webSocketClient != null && webSocketClient.isOpen()) {
        webSocketClient.send("{\"type\":\"ping\"}")
        heartbeatHandler.postDelayed(heartbeatRunnable, 30000)
    }
}
```

### 12. 日志处理不规范

**位置:** `GuardianService.java:242`, `GuardianService.java:327`, `GuardianService.java:379`, `WebSocketService.java:202` 等多处

```java
// ❌ 问题代码
catch (Exception e) {
```

**问题:**

- 错误日志可能包含敏感信息（设备 ID、URL 等）
- 多处重复相同的不规范日志

**修复建议:**
```kotlin
// ✅ 使用 Timber 或其他日志框架
Timber.e(e, "WebSocket 消息解析失败, json=%s", message)

// 或自定义 Log 工具类
LogUtils.e(TAG, "消息解析失败", e) // 统一格式，支持远程上报
```

## ✅ 总体评价

**优点:**
1. 代码结构清晰，分层合理（data/network/service/ui）
2. 网络层使用 Retrofit + OkHttp，架构成熟
3. WebSocket 重连机制实现完善（指数退避）
4. 服务保活使用前台服务 + 通知，符合 Android 规范
5. 日志拦截器设计有亮点（TimingInterceptor）

**主要风险:**
